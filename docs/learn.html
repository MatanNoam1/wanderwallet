1. What the app is

Wanderwallet is a shared travel-expense tracker for two people on a trip. You log spending four ways: type it in the app, snap a receipt photo, or send a photo or a text message to a Telegram bot. A receipt photo or a text like "lunch 240 thb" is read by an LLM, turned into structured data, and dropped into one shared ledger with budget tracking, a category breakdown, and correct per-currency totals.

It is self-hosted on a single small server (a VPS). That one constraint drives most of the architecture: because it is always-on and single process, we avoid a whole class of infrastructure that serverless hosting would have forced on us.

2. Architecture at a glance

Internet (mobile browser + Telegram) | Caddy (auto-HTTPS, 443) | +----------------+------------------------+ | Next.js (one Node process, next start) | | | | React UI -- /api routes -- Auth (Google)| | | | | | capture writes Expense(PROCESSING), | | | enqueues a Job, returns ~200ms| | SSE feed in-process worker loop | | ^ | drains the Job table | | | v | | EventEmitter <- LLM parse + FX convert | +-------+-----------+----------------------+ | | Prisma (one client) storage (local disk) | | SQLite (WAL) /var/lib/.../uploads | | Litestream -> B2 restic -> B2 (nightly)
WHY ONE PROCESS SQLite allows a single writer at a time. If two processes write the same file you get lock errors. By running the web server, the API, the Telegram handler, and the background worker all inside one Node process with one database client, there is exactly one writer and the whole contention problem disappears. This is why no Redis, no separate queue service, and no separate worker process are needed at this scale.

The phases reflect this. P0 was the skeleton (auth, schema, single-process boot). P1, the phase this guide ships with, is the synchronous core: the data model, money math, FX conversion, manual entry, and the dashboard that reads it all back. The harder async machinery (background jobs, live updates, the Telegram bot, vision parsing) is built on top in P2 and P3, reusing the exact same expense shape P1 defines.

WHY THIS ORDER Prove the cheap, deterministic path first. Manual entry exercises the whole data model and money logic with no AI and no async timing involved. Once that is correct and visible end to end, the async paths only have to add "do it in the background" - they do not have to also debug the data model at the same time. Build the risky thing on top of a thing you have already proven.

3. The stack, and why each piece

WHY PORTABLE-BY-DEFAULT The schema avoids SQLite-only tricks: string IDs (cuid) instead of auto-increment, no reliance on a JSON column type, money as plain integers. The payoff: if the app ever outgrows SQLite, moving to PostgreSQL is a config change, not a rewrite. Choosing the small tool now without painting yourself into a corner later is a senior move.

4. Data model and the money problem

The core tables: Trip has many Expenses; each expense is paid by a User with an optional PaymentMethod; a parsed receipt can have many LineItems. Background work lives in a Job table, and daily exchange rates in FxRate.

Never store money as a floating-point number

0.1 + 0.2 is not 0.3 in floating point. For money that is a bug waiting to happen. The fix is to store integer minor units: $5.00 is stored as the integer 500. All math is integer math; you only divide for display.

WHY NOT JUST "CENTS" "Multiply by 100" is wrong for the Japanese yen. The yen has no smaller unit - 1 yen is already the minor unit. Treating ¥1500 as 150000 inflates it 100x. So every currency carries its own decimal count (USD/EUR = 2, JPY = 0), and conversions go through that. See src/lib/money.ts and its tests in src/lib/money.test.ts. 
// src/lib/money.ts
toMinor(12.5, "USD")   // 1250
toMinor(1500, "JPY")   // 1500  (not 150000)
fmt(524000, "USD")     // "$5,240.00"
fmt(1500, "JPY")       // "¥1,500"

Foreign exchange, stored at capture time

A trip has one base currency. An expense in another currency is converted and we store both amounts plus the exact rate used. Storing the rate makes history reproducible and auditable: a rate change tomorrow never silently rewrites what you spent yesterday. Rates are fetched once per day and cached in FxRate; if the network fails we fall back to the last known rate rather than blocking a capture.

5. One request, end to end

The manual-add path, which is all of P1, step by step:

  1. UI - QuickAdd.tsx (a Client Component) collects amount, currency, category, payer, and posts JSON to /api/expenses.
  2. Auth - the route calls requireUser(); no valid session, no entry.
  3. Validation - Zod parses the body. Bad input returns 400 before any database call.
  4. Authorization - requireMember(tripId) confirms this user actually belongs to that trip. Authentication ("who are you") and authorization ("are you allowed") are separate checks.
  5. Money + FX - amount becomes integer minor units; if the currency differs from the trip base, convert using today's rate and record it.
  6. Write - one Expense row, status CONFIRMED (manual entries are trusted, so no background job).
  7. Refresh - the client calls router.refresh(); the Server Component re-queries and re-renders the dashboard with the new total. No manual cache juggling.
The async paths (photo, Telegram) differ only in step 5-6: they save a PROCESSING row, enqueue a Job, return immediately, and a worker later fills in the parsed data and flips the status. Same table, same shape - that is the reuse P1 was designed to set up.

6. Auth and access control

Sign-in is Google OAuth through Auth.js, with sessions stored in the database. Two helpers in src/lib/session.ts are the only gate the rest of the code needs: requireUser() (must be signed in) and requireMember(tripId) (must belong to that trip). Every API route and protected page calls one of them first.

WHY A DEV BYPASS, GATED Google credentials are not wired yet, so a dev-only fallback lets us view the dashboard locally as a seeded user. It is hard-gated to NODE_ENV !== "production", so it can never run on the real server. This is the right way to take a shortcut: make it impossible for the shortcut to reach production, and label it for removal.

7. The professional workflow

Everything below is what separates "code that works on my machine" from "code a team can trust." None of it is heavy. It is a few habits enforced by tooling so they happen every time instead of when you remember.

8. Git: branches, commits, pull requests

Never commit straight to main

main is the always-deployable branch. New work happens on a short-lived branch, then merges back through a pull request (PR). This keeps main green and gives every change a review surface and a revert point. 

git checkout -b p1-expense-core   # branch for the feature
# ... do the work ...
git push -u origin p1-expense-core
gh pr create --fill               # open a PR
# CI runs on the PR; merge when it is green

Conventional Commits

Commit subjects follow a small grammar: type: summary. Common types: feat (new capability), fix (bug), chore (tooling/config), ci, docs, refactor. The subject says what in under ~50 characters; the body explains why when it is not obvious. 

feat: P0 scaffold - Prisma schema, Auth.js, WAL
chore: project docs and CI
ci: add GitHub Actions workflow (lint, typecheck, build)
WHY THE GRAMMAR A consistent prefix makes history scannable, lets tools auto-generate changelogs, and forces you to commit one logical change at a time. If you cannot write a single clean subject, the commit is probably doing too much.

9. CI and branch protection

CI (Continuous Integration) is an automated check that runs on every push and PR. Ours lives in .github/workflows/ci.yml and runs four steps on a clean machine: install dependencies, generate the Prisma client, then lint, typecheck, and build. If any fails, the PR is marked red. 

- run: npm ci            # clean, reproducible install
- run: npx prisma generate
- run: npm run lint      # style + correctness rules
- run: npm run typecheck # tsc --noEmit, no type errors
- run: npm run build     # it actually compiles
WHY "ON A CLEAN MACHINE" "Works on my machine" usually means an uncommitted file or a globally installed tool. CI starts from nothing but the repo, so it catches the thing you forgot to commit. Green CI means a fresh clone will build.

Branch protection

A rule on main turns the workflow from "please remember" into "cannot skip." On this repo: a PR is required to merge, the build check must be green, history must stay linear, and force-pushes and deletes are blocked. The CI run on dummy environment variables proves the build needs no real secrets to compile.

10. Testing and verification

Not everything needs a test, but money math does - it is exact logic with edge cases. money.test.ts uses Node's built-in test runner (no framework to install) to lock in the rules that matter: yen is not treated as cents, floating-point dust is rounded away, conversions respect the target currency's decimals.

WHY VERIFY, NOT ASSUME P1 was not called "done" because it compiled. It was run: seed the database, start the server, load the dashboard, add a real expense, and confirm the total moved from $5,212.20 to $5,284.70 and the breakdown updated. A change is finished when you have watched it work, not when you expect it to.

11. Deployment and ops

ConcernToolRole
ProcesssystemdRuns next start as a non-root user, restarts on crash, starts on boot, caps memory.
HTTPS + proxyCaddyAutomatic TLS certificates and reverse-proxy to the local Node port. Simpler than Nginx for one service.
DB backupLitestreamContinuously streams the SQLite file to cloud storage, so a dead server rebuilds with near-zero data loss.
File backupresticNightly encrypted backup of uploaded receipt images.
WHY A SINGLE WRITER STILL NEEDS WAL SQLite's WAL (write-ahead logging) mode lets readers and the writer work at the same time without blocking, and busy_timeout makes a contended write wait-and-retry instead of erroring. Both are set once at boot in instrumentation.ts. Small database, but still tuned for correctness under concurrent reads.

12. Reusable checklist for your next project

Before writing features
  • Pick the smallest infrastructure that fits the real scale, but keep the schema portable.
  • Decide the data model and the "hard" representations (money, dates, currency) up front.
  • Initialize Git, add a real README and a license, set up .gitignore so secrets and local DBs can never be committed.
While building
  • One feature per branch, one logical change per commit, conventional commit messages.
  • Validate every untrusted input at the boundary; separate authentication from authorization.
  • Prove the deterministic path before the async or AI path.
  • Leave one runnable test behind any non-trivial logic (especially money and security).
Before merging
  • CI green on a clean machine: lint, typecheck, build.
  • Branch protection so the rules cannot be skipped.
  • Actually run the app and watch the change work, then merge via PR.
Before shipping to a server
  • Process manager with auto-restart, HTTPS, least-privilege user.
  • Automated, tested backups - an untested backup is not a backup.
  • Never log secrets, raw user messages, or file contents.
